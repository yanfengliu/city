# Pedestrians and retail activity

## Scope

One citizen entity remains one household of exactly three named people. A visible pedestrian is one stable member performing the household's current activity, not a decorative agent and not an extra population entity. The bounded activity cycle is `home → work → home → free time → home`; work may be in commercial or industrial buildings, while free time is a weighted choice among shopping, leisure, and rest. Household wallets, store inventory, freight, an industrial production chain, and independently concurrent schedules for all three members remain deferred.

The person identity used across simulation, protocol, rendering, selection, and inspection is `(citizen entity id, citizen generation, household-local member id)`. A household owns one employment slot and at most one walker or car at a time, so the selected named resident can remain stable while another member becomes the active traveller; this is deeper identity and biography within the existing household-scale simulation, not a claim that three independent citizen entities now exist.

## Persistent identity and biography

Every new household receives a deterministic `citizenProfile` component with a family name and exactly three members. Each member has a stable id, given name, age, life stage, education, and current role; the one primary worker's role follows the household's commercial or industrial job assignment. Profile generation hashes the city seed, citizen id and generation, and home id without consuming `world.random()`, so identity cannot perturb growth, trip, or replay outcomes.

The cold `citizenProfile` and `citizenLife` components are separated from the hot per-tick `citizen` activity component. Profiles change only when a role changes, and biography changes only when one of the newest eight meaningful events is appended: move-in, hire, job loss, outing departure, or stranding. Reads and appends sanitize malformed legacy arrays without throwing; dropping an invalid or ninth-oldest entry marks the biography truncated and advances its earliest supported tick, so the panel never presents a bounded tail as a complete life. An `outingDeparted` event proves that a named member left for an activity; it deliberately does not claim that the trip completed. Event places carry an entity generation when the place was live, guarding the record against id reuse.

Snapshots created before these components existed remain loadable. Inspection derives the same fallback profile and current employment role from live simulation identity without inventing a past, labels its source `legacyFallback`, and reports that earlier life events are unavailable; the cold components are added only when a later real change needs to be persisted. An in-flight legacy pedestrian path without a stored member projects the household's valid current traveller or deterministic primary-member fallback, so it remains selectable immediately after restore.

## Deterministic activity state

The citizen component keeps the logical phase, optional next activity/shop assignment, and the household-local id of the member representing the current activity. A moving walker is a separate transient entity owned by that citizen. Its `pedestrianPath` component contains the immutable road-cell path, citizen and destination generations, stable member id, purpose, and outbound/return direction. Its small `pedestrian` component contains only segment index and fractional progress, so each tick dirties motion data without copying the full path or cold identity/history into the recorder diff.

The trip system considers a bounded, rotating set of eligible households. Short work commutes walk; longer work commutes retain graph-routed vehicles. Shopping always walks to the nearest eligible commercial building in the same road component, leisure prefers a reachable nearby park and otherwise visits one of the nearest shops, and rest spawns no moving agent. The primary worker owns work trips, another adult owns shopping when one exists, the youngest member owns leisure, and a senior owns rest when one exists, all by deterministic rules. Household composition has a gameplay consequence: each child or teen increases leisure weight and each senior increases rest weight. All stochastic choices still use `world.random()` and all entity/candidate scans have explicit stable ordering.

## Exact road access

Pedestrians route between the actual first road cells adjacent to each building footprint. They use a topology-versioned road-cell path cache with deterministic four-neighbor order. This is intentionally separate from vehicle graph routing: resolving an interior road cell to one compressed edge endpoint can collapse two buildings on the same long street to the same graph node and make a trip teleport.

Road topology changes clear the pedestrian cache and validate every remaining path cell. A route that references a removed road is cancelled, increments `disconnectedTrips`, restores the citizen to the logical origin phase with retry backoff, and never credits a retail visit. On every ordinary movement tick, walkers sharing one directed road-cell segment process leader-first and followers clamp forward progress to preserve a 0.25-cell personal-space gap without overtaking.

## Valid destinations and settlement

A work arrival is valid only while the citizen still owns that work assignment and the destination entity generation matches. Newly spawned vehicles store that destination identity; missing or partial identity metadata from a legacy in-flight vehicle fails closed to a home/work retry, and unassigning a workplace synchronously retires every owned vehicle. A shopping destination must be a live, staffed, powered, watered commercial building with a matching generation. The shopping assignment is stored through the at-shop wait so the return leg cannot accidentally target a recycled entity id.

Only a valid outbound shopping arrival increments `pendingRetailVisits` and `completedShoppingTrips`. The budget system converts pending visits into commercial retail tax as `visits × spendPerVisit × commercialTaxRate / 100`, reports that amount separately as `retailIncome`, includes it in total income, and clears the pending count exactly once. Population-based commercial demand remains as the bootstrap path so a new city does not require completed shopping trips before its first shop can grow.

## Selection and on-demand inspection

The worker's per-tick pedestrian projection includes the current segment plus the stable citizen generation and member id. The main thread raycasts an invisible person-height cylinder around each tiny walker rather than demanding a pixel-perfect hit on the visible silhouette; explicit crowd bounds are refreshed with instance matrices so a bounds cache first computed for an empty crowd cannot make later pedestrians unselectable. Where forgiving cylinders overlap, projected center distance chooses the person closest to pointer intent rather than whichever cylinder surface is first. Buildings, services, and the solid coal/wind/pump models block through-wall selection, while thin poles/wires/pipes do not; a 100 ms stationary-pointer recheck follows walkers and camera movement without raycasting every render frame. Pointer hover and selection use persistent rings keyed to the complete person identity, and a selected person remains selected even while no walker for that member is active.

Clicking a walker sends a generation-guarded `inspectCitizen` query. Clicking “Meet a resident” on an occupied residential building sends `inspectHomeResident`, which queries residents on demand and cycles through every member in canonical household/member order without adding all resident ids to normal building diffs; its cursor carries citizen id, citizen generation, and member id so reuse cannot skip or impersonate a person. Citizen detail is refreshed no more often than every 500 ms of wall-clock time while the panel is open, independent of simulation speed; request ids and entity generations reject late, stale, or recycled-identity replies.

The returned detail names the selected resident, all three household members, the activity representative (called an active traveller only while an agent really exists), current status, happiness and its strongest factors, home, work, destination, active outing venue, commute, and the bounded recent biography with completeness/truncation metadata. Map evidence outlines home in cyan and work in orange; the general generation-checked building-or-service `destinationPlace` drives magenta while travelling—including park trips and the return home—while `activityPlace` retains park/shop provenance through the return leg and drives magenta only while dwelling. The scrollable panel measures itself below the wrapped top HUD, preserves its action button and keyboard focus during 500 ms refreshes, and reconciles the residential cycle total against live home occupancy. The bounded `render_game_to_text()` inspect branch exposes the same selected identity, profile, activity, anchors, active agent, life events, and residential-cycle context for headless verification.

## Worker and renderer boundary

The worker projects the full active pedestrian list each tick as `{id, generation, citizen, citizenGeneration, memberId, fromCell, toCell, t, purpose, outbound}`. Long paths and household profiles stay inside the worker; a missing legacy path member is replaced at projection time by the valid current household traveller or deterministic primary member. The renderer interpolates one current road-cell segment, offsets opposing directions onto separate curbside sidewalk lanes, and samples the shared terrain surface. A stable `(citizen, citizenGeneration, memberId)` hash keeps a named person's garments, skin tone, body proportions, and gait consistent across transient walker entities, return legs, and later outings.

Pedestrians use seven fixed-capacity instanced batches: three for the rigid body (tops, lower garments, heads) and four for the swinging limbs (left/right legs, left/right arms), with a small front cue for facing. Limb geometry is authored with the joint at the origin so an instance matrix can rotate it about hip or shoulder; the walk cycle is a pure function of distance travelled, so a paused game holds its pose and no clock is involved. They do not cast shadows, so moving agents do not invalidate the cached 2048-square shadow map. Empty-list cleanup and entity-generation replacement are explicit, and the global simulation/render cap bounds both ECS and GPU work.
